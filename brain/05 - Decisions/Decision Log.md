# Decision Log

All significant architectural and product decisions made during development. Each entry explains what was decided, why, and what alternatives were considered.

| Date | Decision | Reasoning | Alternatives Considered | Made By |
|------|----------|-----------|------------------------|---------|
| 2026-05-01 | **REST API over GraphQL** | REST is simpler to implement, cache, debug, and version for MVP. GraphQL overkill until data access patterns are well-understood. | GraphQL, tRPC | Architect |
| 2026-05-01 | **React Native + Expo over Flutter** | React Native has larger talent pool, TypeScript ecosystem shared with backend, better third-party library coverage for CRM-specific use cases (calendars, contacts). Expo removes native build complexity. | Flutter, Expo bare workflow | Architect |
| 2026-05-01 | **Expo Router (file-based nav) over React Navigation** | File-system routing convention removes manual route registration. Typed routes and deep linking come for free. Better alignment with the TypeScript-first monorepo setup. | React Navigation v6, Expo bare + React Navigation | Architect |
| 2026-05-01 | **Zustand over Redux Toolkit** | Redux Toolkit is over-engineered for the actual volume of local state this app needs. Zustand is ~1KB, zero boilerplate, and handles auth tokens, UI state, and user preferences with simple stores. RTK was not earning its weight. | Redux Toolkit, Jotai, Valtio | Architect |
| 2026-05-01 | **React Query for server state + offline** | React Query's `networkMode: 'offlineFirst'` handles caching, background refetch, optimistic updates, and offline mutation queuing in one library. Eliminates the need for a custom offline mutation queue + MMKV sync engine. Simpler than RTK Query + manual offline logic. | RTK Query, TanStack Query standalone, SWR, custom fetch layer | Architect |
| 2026-05-01 | **Fastify over Express** | Fastify is 2–3x faster than Express in benchmarks, has built-in JSON schema validation hooks, is WebSocket-compatible via `@fastify/websocket`, and has a better TypeScript story. Railway (persistent server) supports long-lived connections Fastify needs. | Express, NestJS, Hono, Koa | Architect |
| 2026-05-01 | **Prisma ORM over Drizzle** | Prisma's Supabase PostgreSQL integration is mature and well-documented. `prisma migrate` is more ergonomic than Drizzle's migration story. Strong autocomplete + type inference from generated client. Team velocity benefit outweighs the bundle size difference. | Drizzle ORM, raw `pg`, Knex | Architect |
| 2026-05-01 | **Supabase (hosted Postgres + Auth + Realtime) over self-managed stack** | Supabase replaces five separate infrastructure components (PostgreSQL, PgBouncer, Redis, custom JWT auth, Socket.io) with one managed service. RLS still works identically via `auth.uid()`. Estimated infra cost drops to ~$5/month. No DevOps overhead for MVP. | Self-managed PostgreSQL + Redis + custom auth + Socket.io, Firebase (rejected — see below), PlanetScale | Architect |
| 2026-05-01 | **Supabase Realtime over Socket.io** | Supabase Realtime is included in the Supabase stack and covers all real-time requirements: live deal board updates, team presence, inbound message delivery. Eliminates a separate WebSocket server entirely. | Socket.io, Pusher, Ably | Architect |
| 2026-05-01 | **Supabase Auth over custom JWT** | Supabase Auth handles JWT issuance, refresh, OAuth, magic links, and session management. Eliminates custom bcrypt + refresh_tokens table + JWT signing code. RLS policies automatically have access to `auth.uid()`. | Custom JWT + bcrypt, Auth0, Clerk | Architect |
| 2026-05-01 | **Railway (not Vercel) for API hosting** | Vercel is serverless — it cannot hold persistent WebSocket connections, which are required for Twilio webhook reliability and any server-side real-time logic. Railway runs a persistent Node.js process, supports WebSockets, and costs ~$5/month. **Vercel is definitively eliminated for the backend.** | Vercel (eliminated — serverless/WebSocket incompatible), Render, Fly.io, AWS ECS | Architect |
| 2026-05-01 | **Firebase ruled out in favor of Supabase** | Firestore is a document DB poorly suited for relational CRM data (contacts → deals → pipeline stages → tasks is a multi-table join, not a document). PostgreSQL's relational model and Supabase's RLS are the right fit. Firebase Auth is also less ergonomic for custom claims and multi-tenant RLS. | Firebase (Firestore + Firebase Auth) | Architect |
| 2026-05-01 | **AWS/GCP ruled out for MVP** | Operational overhead (VPC, IAM, ECS/EKS, RDS, ElastiCache) requires DevOps expertise we don't have or need at this stage. Railway + Supabase delivers the same result with zero ops burden. Revisit when monthly revenue exceeds ~$50K. | AWS ECS + RDS + ElastiCache, GCP Cloud Run | Architect |
| 2026-05-01 | **Expo Push Notifications over direct FCM/APNS** | Expo's push notification service wraps both FCM and APNS behind a single API. No separate FCM credentials or APNS certificates needed during development. The Expo service forwards to the native push providers. Production apps can later bypass Expo for direct delivery. | Direct FCM + APNS, OneSignal | Architect |
| 2026-05-01 | **Twilio for SMS over custom telephony** | Twilio is the industry standard for programmable SMS; webhook delivery, message status tracking, and global number support out of the box. | Vonage, MessageBird, AWS SNS | Product |
| 2026-05-01 | **Expo EAS Build + EAS Update** | Cloud builds remove the requirement for local Xcode/Android Studio. OTA updates allow hotfixes without App Store review cycles. | Manual local builds, CodePush, Custom OTA | Architect |
| 2026-05-01 | **No freemium tier — 14-day free trial instead** | Freemium requires maintaining a free tier indefinitely and risks subsidizing non-paying users who never convert. 14-day full-featured trial converts at higher rates and sets clearer expectations. | Freemium free forever, 30-day trial, credit card required | Product |
| 2026-05-01 | **14 features in MVP (not phased features)** | All 14 features are interdependent — contacts without pipeline have less value; pipeline without analytics misses insights; messaging without history loses context. Launching with all 14 creates the coherent product experience the market positioning requires. | Launch with 5 core features, add others in subsequent releases | Product |
| 2026-05-01 | **Google Vision API for business card OCR (not on-device)** | On-device ML requires model management, larger app size, and achieves lower accuracy. Google Vision API is best-in-class for structured document text extraction. Privacy trade-off acceptable since user initiates the capture explicitly. | Core ML (iOS), TensorFlow Lite, AWS Textract | Architect |
| 2026-05-01 | **Shared Zod schemas across mobile and backend** | Same Zod schema files imported by Fastify (request validation) and React Native (form validation, response parsing). One source of truth for API contracts eliminates a whole class of frontend/backend type divergence bugs. | Separate schema definitions per layer, OpenAPI codegen | Architect |
| 2026-05-01 | **Multi-tenant via RLS (not separate schemas)** | Row-Level Security in PostgreSQL (via Supabase) enforces tenant isolation at DB layer, not application layer. Simpler ops than schema-per-tenant. Scales to thousands of orgs in one database. | Schema-per-tenant, separate databases per tenant | Architect |
| 2026-05-02 | **Fastify upgraded to v5 (was v4 in scaffold)** | Session 1 scaffold specified Fastify v4, but `fastify-type-provider-zod@4.x`, `@fastify/cors@11.x`, `@fastify/jwt@9.x` all require Fastify v5. Aligning to v5 is the correct long-term direction; v4 is already past its active support window. All `@fastify/*` plugins bumped to their v5-compatible major versions. | Stay on v4 + downgrade type provider (rejected — type provider v2.x has fewer features) | Architect (Sprint 0) |
| 2026-05-02 | **`async start()` wrapper in backend/index.ts (not top-level await)** | Root `package.json` cannot have `"type": "module"` because Expo Metro bundler requires CJS for the mobile side. Without ESM mode, `tsx` runs the backend as CJS where top-level `await` is unsupported. Wrapping startup in `async start(){}; start()` is the zero-risk solution — no separate `backend/package.json` needed for MVP. | Add `"type":"module"` to root (rejected — breaks Expo), create `backend/package.json` with ESM (deferred to Sprint 1 if needed) | Architect (Sprint 0) |
| 2026-05-02 | **`--legacy-peer-deps` for npm install** | React Native test library (`@testing-library/react-native@12.x`) has an unresolved peer dep on `react-test-renderer` that conflicts with `react@18.2.0`. This is a pre-existing scaffold issue unrelated to backend work. `--legacy-peer-deps` unblocks installs without changing any package versions. Document in CLAUDE.md that install must use this flag. | `npm install --force` (rejected — modifies lockfile aggressively), upgrade React/RN (deferred — major scope change) | Architect (Sprint 0) |
| 2026-05-02 | **Auth on Fastify API using bcrypt + @fastify/jwt (not Supabase Auth SDK)** | API Design doc originally said auth lives entirely on Supabase Auth (mobile client calls Supabase directly; Fastify only receives pre-verified JWTs). However: (1) the session prompt explicitly asks for register/login endpoints on the Fastify API; (2) the User model has a `password_hash` field — useless if Supabase manages passwords; (3) Phase 1 only specifies DATABASE_URL + JWT_SECRET, not SUPABASE_URL/SERVICE_ROLE_KEY; (4) @fastify/jwt is already registered in index.ts. Decision: auth endpoints live on Fastify; bcryptjs handles password hashing; @fastify/jwt signs tokens. Supabase Auth SDK can be layered in for OAuth/magic links in a later sprint. | Supabase Auth SDK (rejected for Sprint 1 — requires additional env vars not in scope), pass-through Supabase JWT (rejected — requires JWKS verification, different from @fastify/jwt setup) | Architect (Sprint 1) |
| 2026-05-02 | **Soft delete = status → 'archived' (no deleted_at column)** | Session prompt says "soft delete (set deleted_at)" but the contacts table has no `deleted_at` column — neither in the SQL schema nor in the Prisma model. The existing pattern is `status = 'archived'` (used for contact archive, deal archive). Using `deleted_at` would require a schema migration and change the established pattern. Implementing `status = 'archived'` is consistent with the rest of the schema. | Add deleted_at column (deferred — schema migration needed, inconsistent with existing archive pattern) | Architect (Sprint 1) |
| 2026-05-02 | **App-level org scoping on all Prisma queries** | Prisma connects as the Supabase database superuser, which bypasses Row-Level Security. All controllers must add `where: { organization_id: request.user.org_id }` to every query. RLS in the SQL schema is a safety net for the Supabase client SDK, not for Prisma. | Rely on RLS only (rejected — Prisma bypasses RLS with service connection), use Prisma middleware to inject org scope (deferred — adds complexity for MVP) | Architect (Sprint 1) |

## Superseded Decisions

These earlier decisions were reversed when the stack was finalized:

| Original Decision | Superseded By | Date |
|-------------------|--------------|------|
| Express over NestJS | **Fastify** chosen — faster, WebSocket-compatible, better TypeScript | 2026-05-01 |
| Drizzle ORM over Prisma | **Prisma** chosen — better Supabase integration, more ergonomic migrations | 2026-05-01 |
| Self-managed PostgreSQL | **Supabase hosted PostgreSQL** — eliminates ops burden | 2026-05-01 |
| Redis for caching/sessions | **Supabase Auth + Supabase Realtime** replaces Redis for these use cases | 2026-05-01 |
| Bull + Redis for job queues | **Partially superseded** — Supabase Realtime handles the real-time use cases Bull was solving; async job needs (PDF export, bulk import) remain TBD | 2026-05-01 |
| Socket.io for WebSocket | **Supabase Realtime** — included in Supabase stack, no separate server needed | 2026-05-01 |
| Railway/Render (either/or) | **Railway specifically** chosen — Render doesn't support persistent WebSocket connections on free tier; Railway confirmed | 2026-05-01 |
| Redux Toolkit | **Zustand** — less boilerplate, sufficient for actual local state needs | 2026-05-01 |
| React Navigation v6 | **Expo Router** — file-based routing, typed routes, better DX | 2026-05-01 |
